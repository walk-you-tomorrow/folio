## 🔴 Critical — 개발 전 반드시 수정

### C-1. Liveblocks 사용 시 REST API 엘리먼트 CRUD 이중화 문제 (Developer)
- **문제:** API 명세에 엘리먼트 CRUD REST 엔드포인트가 있고, 동시에 Liveblocks Storage로도 엘리먼트를 관리한다. 둘 다 존재하면 데이터 정합성이 깨진다.
- **결정:** Liveblocks Storage를 실시간 엘리먼트의 단일 소스(Single Source of Truth)로 사용. REST API는 캔버스 생성/조회/삭제만 담당. 엘리먼트 CRUD는 Liveblocks를 통해서만 처리.
- **영향:** API 명세서에서 엘리먼트 관련 REST 엔드포인트 제거 필요.

### C-2. 캔버스 최초 로드 시 데이터 출처 불명확 (QA)
- **문제:** 캔버스 접속 시 GET `/api/canvas/:id`로 엘리먼트를 조회하는데, Liveblocks Storage가 소스라면 이 API가 어떻게 엘리먼트를 반환하는지 불명확.
- **결정:** 캔버스 메타데이터(id, title, expires_at)만 REST로 조회. 엘리먼트는 Liveblocks Room 입장 시 자동 로드. GET `/api/canvas/:id`에서 elements 필드 제거.

### C-3. 캔버스 만료 후 데이터 정리 메커니즘 없음 (Developer)
- **문제:** 캔버스 수명 7일이지만, Liveblocks Room과 Supabase Storage 이미지를 정리하는 로직이 없다.
- **결정:** Supabase의 pg_cron 또는 Vercel Cron으로 만료 캔버스 일괄 정리 작업 추가. Liveblocks Room은 API로 삭제.

## 🟡 Warning — 수정 권장

### W-1. nanoid 8자리 충돌 가능성 (Developer)
- **문제:** nanoid 8자리(기본 알파벳)의 조합 수는 약 2.8조개로 충분하지만, 충돌 검사 로직이 명세에 없다.
- **권장:** 생성 시 DB unique constraint + 재시도 로직 1줄 추가.

### W-2. 캔버스 생성자와 방문자의 플로우 차이 미정의 (QA)
- **문제:** 캔버스를 만든 사람에게는 공유 모달이 자동 표시되지만, 이후 재방문 시에도 모달이 뜨는지, 닉네임 입력이 필요한지 미정의.
- **권장:** 생성자는 sessionStorage로 첫 방문 여부 판단. 재방문 시 모달 미표시.

### W-3. 화면 정의서에 모바일 툴바 상세 없음 (Designer)
- **문제:** 모바일에서 하단 툴바가 키보드와 겹칠 수 있음. 터치 드래그와 캔버스 패닝 구분 미정의.
- **권장:** 모바일에서는 모드 전환 방식 명확히: 기본=패닝, 도구 선택 시=편집 모드.

### W-4. 동시 편집 시 같은 스티커 수정 충돌 UX 미정의 (QA)
- **문제:** Last-Write-Wins 정책이지만, 두 사용자가 같은 스티커를 동시에 편집할 때 UI 피드백이 없으면 혼란.
- **권장:** "OO님이 수정 중" 표시가 화면 정의서에 있으므로, 수정 중인 스티커는 다른 사용자의 편집을 잠금(soft lock) 처리.

### W-5. 이미지 업로드 동시 제한 미정의 (QA)
- **문제:** 여러 이미지를 동시에 드래그앤드롭할 때 동시 업로드 수 제한이 없다.
- **권장:** 동시 업로드 최대 5개, 초과 시 큐잉. 캔버스당 이미지 총 용량 제한 (50MB) 추가.

### W-6. 접속자 아바타 컬러 할당 로직 없음 (Designer)
- **문제:** 접속자마다 고유 컬러 아바타를 표시하는데, 컬러 할당 방식이 정의되지 않음.
- **권장:** 프리셋 8색 배열에서 입장 순서대로 할당. 브랜드 팔레트 기반 컬러 세트 정의 필요.

## 🟢 Good — 잘 되어 있는 부분

### G-1. 브랜드 정합성 (Designer) ✅
- 화면 정의서의 컬러, 폰트, 모서리 반경, 그림자, CTA 규칙이 브랜드 가이드라인과 완벽히 일치.
- 마이크로카피가 브랜드 보이스(해요체, 솔직, 담담)를 정확히 따름.
- Primary CTA 한 화면 1개 원칙 준수.

### G-2. 수용 기준 테스트 가능성 (QA) ✅
- 각 기능별 수용 기준이 체크리스트 형태로 명확. 자동화 테스트 케이스로 전환 가능.

### G-3. 에러 메시지 (QA + Designer) ✅
- API 에러 응답이 브랜드 톤앤매너(해요체)로 작성. 개발자 코드 노출 없음.
- 404, 410, 413, 429 등 HTTP 상태별 사용자 친화적 메시지 정의.

### G-4. 데이터 모델 간결성 (Developer) ✅
- Canvas, Element, Session 3테이블로 최소한. JSONB content로 타입별 확장 가능.

### G-5. 키보드 단축키 정의 (Designer) ✅
- 접근성 + 파워유저 양쪽을 고려한 단축키 세트.

### G-6. 마이크로 인터랙션 명세 (Designer) ✅
- 애니메이션 종류, 타이밍, easing이 구체적으로 정의되어 개발 시 모호함 없음.

## 📝 Suggestion — 제안사항

### S-1. OG 메타태그 & 링크 프리뷰 (Developer)
- 카톡/슬랙 공유 시 링크 프리뷰(OG Image, title, description)가 있으면 클릭률이 올라감.
- 캔버스별 동적 OG 이미지 생성 권장 (MVP에서는 정적 기본 이미지라도).

### S-2. 오프라인 모드 고려 (Developer)
- MVP 범위 밖이지만, 네트워크 끊김 시 로컬 변경을 큐에 쌓아두고 재연결 시 동기화하면 UX 크게 향상.

### S-3. 캔버스 스냅샷/썸네일 (Designer)
- 재방문 시 "내 캔버스 목록"에 썸네일이 있으면 좋지만 MVP 범위 밖. Later 기능으로 기록.

### S-4. 성능 테스트 시나리오 구체화 필요 (QA)
- "엘리먼트 100개 + 사용자 10명 동시 접속" 같은 구체적 부하 시나리오 정의 필요.
